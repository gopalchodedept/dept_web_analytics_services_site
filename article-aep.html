How The Adobe Stack
Actually Works.

AEP explained without the acronym soup: what the platform really is, why XDM is the data layer of the Adobe world, how data gets out of an AEM site, and how the Web SDK ties the whole thing together. Written for someone smart who has never touched it.

Entry 001~14 min readAEP · XDM · Web SDK · AEM · CJABy the Web Analytics team

TL;DR

AEP is Adobe's central customer data platform: everything flows in through one pipe (the Web SDK), gets validated against one dictionary (XDM schemas), lands in one place (the Data Lake + Real-Time Customer Profile), and every Adobe app (CJA, RT-CDP, AJO) reads from there.

The build order never changes: schema → dataset → datastream → Tags property → Web SDK → validate. Skipping ahead is how projects die.

On AEM sites, the Adobe Client Data Layer is already built into the core components. You switch it on, listen to it in Tags, and map it to XDM.

01What Is AEP, Really?

Strip away the branding and Adobe Experience Platform is three things: a pipe, a warehouse, and a control tower. Think of it like an airport. Data arrives from everywhere (your website, your app, your CRM, file uploads), the Edge Network is air traffic control deciding where each arrival goes, customs (governance) checks that everything entering is allowed to be there, and the terminals are the Adobe apps where the data actually gets used: Customer Journey Analytics for analysis, Real-Time CDP for audiences, Journey Optimizer for messaging.

The key mental shift from the GA4 world: in Adobe land, the apps don't collect their own data anymore. Collection happens once, centrally, into AEP. The apps are just different windows onto the same data.

02XDM: The Data Layer Of The Adobe World

On the Google side, the data layer is a JavaScript object your site pushes events into. On the Adobe side, the equivalent concept is XDM (Experience Data Model), and the difference is discipline: XDM is a contract enforced by the platform itself. Every event you send must match a schema you defined up front, or it bounces.

Think of a schema as an official form template. It's built from a class (what kind of form: XDM ExperienceEvent for things that happen, XDM Individual Profile for facts about a person) plus field groups (reusable sections of the form: commerce fields, web details, identity fields, your own custom group). A dataset is the filing cabinet where completed forms of one template get stored.

  • Schema = the form template. Defines every field, its type, its rules.
  • Class = what kind of form it is. Event (something happened) or Profile (someone is).
  • Field group = reusable sections. Adobe ships standard ones; you add a custom group for anything unique to the business.
  • Dataset = where the completed forms are filed, one schema per dataset.
  • identityMap = the field that says who this event belongs to (ECID, email, CRM id), with one marked as primary.

Nuance that bites people

Schema changes are additive only. Once a dataset has ingested data, you can add fields but never delete or retype them. Design the schema like it's going in a museum, because effectively it is. This is why the measurement framework and the data architect come before any code.

03Getting Data Out Of An AEM Site

Good news if the site runs on AEM: the data layer is already built. AEM Core Components ship with the Adobe Client Data Layer (ACDL), an event-driven JavaScript data layer maintained by Adobe. You don't write it, you switch it on in the page template policy, and every core component starts announcing itself: pages push page info, buttons push click events, forms push form events, all into a global adobeDataLayer array.

The pattern will feel familiar from GTM land: components push() events, and your tag management listens. The Adobe twist is that the listener lives in Tags (the product formerly known as Launch), where a rule fires on a data layer event, a data element maps the pushed values into an XDM object, and the Web SDK sends it to the Edge.

  1. Enable the data layer in the AEM page template policy (Core Components v2+, it's a checkbox, on by default in recent versions).
  2. Check what's already being pushed. Open the console and run adobeDataLayer.getState(): page, component, and interaction data is usually already there.
  3. Add custom pushes for anything business-specific (search terms, product data, form outcomes). This is a small AEM dev task with a clear spec from us.
  4. Listen in Tags using the Adobe Client Data Layer extension: rules fire on specific data layer events.
  5. Map to XDM in a data element, then send with the Web SDK action. The push's plain names become schema fields.

04The Web SDK: One Library To Rule Them All

Here's the part that genuinely simplified Adobe implementations. The old world required a separate library per product: AppMeasurement.js for Analytics, VisitorAPI.js for identity, at.js for Target, dil.js for Audience Manager, each with its own quirks, race conditions, and page weight. The Web SDK (alloy.js) replaces all of them with a single library that sends one XDM stream to the Edge Network, and a server-side config called a datastream decides which Adobe services receive it.

The setup order matters, and it's the same on every engagement. This is the checklist we run:

  1. Design the XDM schema (ExperienceEvent class + field groups + custom group), driven by the measurement framework.
  2. Create the dataset against that schema. Decide now whether it feeds the Real-Time Customer Profile.
  3. Create the datastream in the Data Collection UI, one per environment (dev, stage, prod), and toggle the services it should feed: AEP, Adobe Analytics (mapped report suite), Target.
  4. Create the Tags property and install the Web SDK extension. Point each Tags environment at the matching datastream environment.
  5. Build data elements for the XDM object, reading from the data layer.
  6. Build rules: on page load and on data layer events, fire sendEvent with the XDM payload.
  7. Set up consent and identity: defaultConsent (usually "pending" in Europe, "in" elsewhere, legal decides), and a first-party CNAME domain so the Edge is called on your own subdomain.
  8. Validate everything with the Adobe Experience Platform Debugger extension and an Assurance session before any launch.

05The Nuances Nobody Tells You

Datastreams are per environment

Dev, stage, and prod each get their own datastream id. Mixing them up is the classic "why is test data in prod" incident. We name them loudly and map them in the Tags extension config, never hard-coded.

Enabling Profile is a commitment

Marking a schema and dataset for the Real-Time Customer Profile changes how data is stored and merged. Do it deliberately, with an identity strategy, not because the toggle was there.

Consent defaults change what fires

With defaultConsent: "pending", the Web SDK queues events until consent arrives; nothing is sent for visitors who never consent. Reporting will look "down" versus an old setup that fired regardless. That's not a bug, that's compliance.

CJA doesn't read your mind

Data landing in the lake isn't automatically in Customer Journey Analytics. Someone builds a connection (choosing datasets) and a data view (defining metrics, dimensions, and session logic). That data view is where "what counts as a visit" actually gets decided.

ACDL events have timing

The data layer is event-driven: if a Tags rule listens for an event that fired before the library loaded, ACDL replays state, but ordering assumptions still bite. We QA event timing explicitly on slow connections.

06Who Does This Work?

Four roles show up on a real AEP build. On smaller engagements one person wears two hats, but the hats are still distinct.

Role 01

AEP Implementation Engineer

Owns
  • Datastreams, Tags property, Web SDK config
  • XDM data elements & sendEvent rules
  • Debugging with AEP Debugger & Assurance
Experience to do the job
  • 2+ years in tag management (GTM absolutely counts)
  • Solid JavaScript and browser devtools fluency
To start learning
  • HTML/CSS/JS fundamentals plus any tag manager
  • Experience League's free "Implement with Web SDK" path
Role 02

Data Architect (XDM)

Owns
  • Schema & field group design, identity strategy
  • Datasets, Profile enablement, governance labels
Experience to do the job
  • 3+ years in data modeling or analytics engineering
  • Comfort with JSON and warehouse concepts
To start learning
  • Strong data modeling basics from any warehouse world
  • Experience League's XDM and Real-Time Profile courses
Role 03

AEM Developer (ACDL side)

Owns
  • Enabling the data layer in template policies
  • Custom adobeDataLayer.push() events from our spec
Experience to do the job
  • AEM Core Components, HTL, and front-end experience
To start learning
  • AEM developer basics plus the ACDL docs on GitHub
Role 04

CJA Analyst

Owns
  • Connections and data views in CJA
  • Workspace dashboards the business actually reads
Experience to do the job
  • 2+ years in digital analytics (Adobe Analytics or GA4)
  • SQL is a welcome bonus, not a requirement
To start learning
  • Analytics fundamentals from either platform
  • Experience League's CJA essentials course
