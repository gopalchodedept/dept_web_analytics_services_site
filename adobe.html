What this unlocks

Three Use Cases

The Adobe stack is a pipeline, and these are the three moments where it pays off, each one building on the last.

Use case 01, the foundation

Stitch Every Source Into One Profile

The same customer shows up as a cookie on the website, an email in the CRM, a device ID in the app, and a loyalty number in store. Until those are connected, every report undercounts and every campaign re-targets people who already converted.

AEP's Identity Service joins them through identity namespaces we define in the schema design, building an identity graph that resolves all of it into a single, real-time customer profile.

  • Identity namespace strategy (ECID, email hash, CRM ID, device ID…)
  • Ingestion of web, app, CRM & offline datasets into AEP
  • Profile-enabled datasets feeding one stitched view per person
Identity graph, four namespaces, one person
Use case 02, the insight

Build Dashboards In CJA

With stitched data flowing in, CJA becomes the single reporting surface for the whole journey. We build the connection (which datasets join together), design the data views (the metrics and dimensions each team sees), and deliver Analysis Workspace dashboards stakeholders open every Monday.

Because the data is person-based, the questions get better: not "how did the website do?" but "how did the customer do, across web, app, email, and store?"

  • Connections joining every profile-enabled dataset
  • Data views with deliberate session, attribution & component settings
  • Curated Workspace dashboards, scheduled delivery, team enablement
Connection → data view → dashboard
Use case 03, the activation

Personalize With AJO + Real-Time CDP

The same profile that powers the dashboards powers the experience. Real-Time CDP evaluates audiences as behaviour streams in, a visitor can qualify for "high-intent, hasn't purchased" mid-session, and AJO picks the journey up from there.

Because it runs on the Edge Network, the personalized experience can come back to the website in the same request that sent the data. And the loop closes: every personalized touch is measured back in CJA, so the next audience is smarter than the last.

  • Streaming audience qualification in Real-Time CDP
  • AJO journeys: triggers, decisioning, channel orchestration
  • Edge-delivered web personalization, email & push from one journey
  • Impact measured in CJA, a true closed loop
The loop: behaviour → audience → journey → experience
How an Adobe engagement runs

The Adobe
Playbook

The same discipline as our GA migrations, adapted to the Adobe stack. Nine phases, and just like on the Google side, the early requirements work (here, the XDM schema design) is what everything downstream depends on.

01
Phase 01

Discovery & Solution Design

Map the current state, the licensed entitlements, and the target architecture. An Adobe engagement can mean very different things depending on whether the client owns CJA, RT-CDP, AJO, or all three.

What happens here
  • Current-state review: AppMeasurement, Launch, Target, report suites
  • Entitlement check, which AEP apps and services are licensed
  • Business requirements & KPI workshop with stakeholders
  • Target architecture: Web SDK + Edge, sandboxes, environments
Output → Solution design document & engagement roadmap
02
Phase 02, The core of the engagement

XDM Schemas & Requirements

The Adobe equivalent of our data layer specs. XDM is the data model everything in AEP speaks, and like a data layer, it has to be designed before it's collected. We translate business requirements into schemas, field by field.

What happens here
  • ExperienceEvent & Individual Profile schema design, with field groups
  • Identity namespaces defined, the keys that make stitching possible
  • Data layer → XDM mapping requirements, event by event
  • Datasets, profile enablement & data governance labels
Output → XDM schema set + collection requirements document
03
Phase 03

AEP & Datastream Setup

Platform configuration: the schemas go live in AEP, and the datastream is built to route incoming events to every service that needs them, Analytics, AEP, Target, from a single collection point.

What happens here
  • Schemas & datasets created across dev / stage / prod sandboxes
  • Datastream configuration with per-service routing
  • Identity Service & merge policy configuration
  • Consent & privacy settings wired into collection
Output → A configured AEP environment, ready to receive data
04
Phase 04

Web SDK Implementation

The build: the Web SDK (alloy.js) implemented through Tags, sending XDM-shaped events to the Edge Network. For migrations, this is where AppMeasurement and at.js are retired, one library, one beacon, one stream.

What happens here
  • Tags property build: extensions, data elements, rules
  • sendEvent design mapped from the website's data layer
  • Dev alignment & sign-off, same ritual as our GA migrations
  • Legacy library decommissioning plan (no double-counting windows)
Output → Web SDK live on staging, sending spec-compliant XDM
05
Phase 05

QA With Assurance

Validation with Adobe's own tooling. Every XDM payload is inspected against the schema, identities are checked end to end, and nothing graduates to production until the stream is clean.

What happens here
  • Assurance sessions validating events, payloads & ordering
  • Experience Platform Debugger checks across key templates
  • Identity validation, namespaces present and resolving correctly
  • Dataset landing checks: data arriving, mapped, profile-enabled
Output → QA report, a verified collection layer
06
Phase 06

CJA Connections & Data Views

Reporting architecture. The connection decides which datasets join into one journey; the data views decide what each team sees and how sessions and attribution behave. Decisions here shape every number anyone reads later.

What happens here
  • Connection built across web, app, CRM & offline datasets
  • Cross-channel stitching via the identity graph
  • Data views per audience, curated metrics, dimensions, naming
  • Session timeout, attribution & filtering set deliberately
Output → CJA connection + governed data views
07
Phase 07

Dashboards & Reporting

Analysis Workspace projects built for the people who'll use them, executive scorecards, channel deep-dives, funnel and journey analysis, with training so the team can self-serve from day one.

What happens here
  • Workspace dashboards per stakeholder group
  • Journey & fallout analysis across stitched channels
  • Scheduled projects & report delivery
  • Enablement sessions, teach the team to fish
Output → A dashboard suite the business actually opens
08
Phase 08

Activation, RT-CDP → AJO

Data becomes action. Audiences are built in Real-Time CDP on the stitched profile, and AJO journeys activate them, web personalization served via the Edge, email and push orchestrated from the same brain.

What happens here
  • Audience strategy & streaming segment build in RT-CDP
  • AJO journey design: entry events, conditions, decisioning
  • Edge-based web personalization wired through the Web SDK
  • Measurement plan, every journey reads back into CJA
Output → Live personalization, running on the profile
09
Phase 09

Hypercare & Enablement

The stack is live; now it has to stay healthy. We monitor collection and stitching through the hypercare window, tune what real traffic reveals, and hand over documentation for everything we built.

What happens here
  • Collection & profile monitoring through hypercare
  • Audience & journey performance tuning
  • Runbook handover: schemas, datastreams, connections, journeys
  • Roadmap for the next wave of use cases
Output → A documented, self-sustaining Adobe stack
