Questions, answered

FAQ.

The questions we get most often from client teams, account leads, and anyone scoping analytics work. If it isn't answered here, ask the team on Slack.

When should the Web Analytics team get involved in a project?+

As early as possible, ideally the moment a redesign, replatform, or new build is scoped. The single most expensive mistake we see is bringing analytics in after development is done: by then the data layer doesn't exist, dev capacity is gone, and the site launches blind.

If a project touches the website's templates, it touches tracking. Loop us in at discovery.

What exactly is a data layer, and why does this site mention it constantly?+

It's a structured JavaScript object (window.dataLayer) the website uses to describe what's happening, page views, add-to-carts, purchases, in a consistent format. Google Tag Manager reads it and translates each event into GA4 hits and marketing pixels.

It's mentioned constantly because everything we deliver depends on it. Tags built on a broken data layer produce broken data, no matter how well the tags themselves are configured.

What happens to tracking when a website is redesigned?+

By default: it breaks. The data layer is code living inside the site's templates, so when templates are rebuilt on a new CMS or design, it does not carry forward. GTM keeps firing, but against a page structure that no longer matches, events go silent or fire wrong.

That's why the migration playbook exists: re-specify the data layer for the new build, QA it, rebuild the container, and launch with verified tracking.

Do you just move existing tags, or build new tracking too?+

Both models exist. A lift-and-shift migrates exactly what's there today, it's the rare case. Most engagements are migrate-plus-build: everything existing moves over, and we design new tracking for the new site's features, since a redesign always introduces journeys the old setup never measured.

How long does a website migration take?+

It tracks the website build itself. The specs and audit phases front-load before development; data layer QA runs alongside dev sprints; GTM and GA4 work stages in parallel; launch syncs with the site go-live. As a rough shape: small sites run a few weeks of analytics effort, large multi-template builds run several months, spread across the project timeline.

What do you need from a client to get started?+

Access, mainly: GA4 (editor), GTM (publish), the ad platforms where pixels live, the staging environment for the new site, and a line to the development team. Plus any existing tracking documentation, though we routinely start from zero documentation; the audit phase exists for exactly that reason.

Who actually writes the data layer code?+

The development team, either DEPT's own developers when we're building the site, or the client's dev team when they own the codebase. Either way, we write the specification (every event, parameter, type, and sample push), align it with the devs for feasibility, and QA the implementation against the spec. Clear division: they own the code, we own the requirements and the verification.

GA4 or the Adobe stack, how do clients choose?+

We start with a platform evaluation if one is needed, and recommend the right platform for the client's reality rather than a default. As a rule of thumb, GA4 + GTM covers most measurement needs. The Adobe stack (AEP, CJA, RT-CDP, AJO) is recommended only when the client already holds Adobe licenses and has genuine cross-channel, person-level requirements, because it costs a lot, in licensing and in build effort. We work both sides, so the recommendation is honest.

Can we set up GA4 events without a data layer?+

We don't recommend it. Without a data layer, tags have to guess from the page itself: DOM-scraping triggers, CSS selectors, URL rules. Those break silently every time the site's markup changes, so the setup is simply not stable. We'll only go this route when a data layer genuinely cannot be implemented, and even then it's flagged as a temporary, fragile arrangement with a plan to move to a proper data layer.

What's the difference between Adobe Analytics and CJA?+

Adobe Analytics is the legacy web-analytics product, built on report suites and a web-centric data model. Customer Journey Analytics is its successor on the Experience Platform: it reports on any dataset in AEP, web, app, CRM, offline, stitched by identity into person-level journeys. Migrating between them means redesigning the data model (XDM), not just moving settings.

The site already launched and tracking is broken. Too late?+

Not too late, just more urgent. We run the audit against the live site, triage what's recoverable, prioritize revenue-critical tracking (conversions, pixels) first, and rebuild the data layer with the dev team in fast follow. Data lost before the fix can't be recovered, so the sooner we start, the smaller the gap.

Do you support things after launch?+

Yes, every migration ends with a hypercare window: daily monitoring, old-vs-new data comparisons, and rapid fixes for issues only real traffic reveals. Beyond hypercare, ongoing support, new pixel work, and reporting builds are all standard engagements.
